# LLM Wiki

A local Python CLI tool that builds and maintains a persistent, interlinked markdown wiki inside a dedicated [Obsidian](https://obsidian.md/) vault. You feed it raw sources (URLs, PDFs, plain text), and the tool extracts content, sends it to an LLM, and receives proposed wiki pages. You can review the batch, edit pages in your system editor, and approve before anything is written to disk.

Supported LLM Providers:
- **Google Gemini** (Default: `gemini-3-flash-preview`)
- **LM Studio** (Local OpenAI-compatible endpoints)

---

## 🚀 Features

- **Automated Ingestion**: Extracts text from PDFs (with OCR fallback), websites, and text files.
- **Batch Editing**: Reviews proposed pages generated by the LLM before saving them to your vault.
- **Knowledge Graph**: Generates a local, offline interactive `vis.js` knowledge graph showing both explicit links and semantic LLM-inferred edges.
- **Built-in Linter**: Scans for internal broken links, orphan pages, and corrupted wikilinks.
- **Natural Language Queries**: Search and synthesize answers from your vault logically without requiring a vector database.

## 🛠️ Installation

### 1. Requirements

- Python 3.10+
- [Tesseract OCR](https://github.com/tesseract-ocr/tesseract) (Optional, required for scanned PDF ingestion)
- [Obsidian](https://obsidian.md/) (Optional, but recommended for viewing the vault)

### 2. Setup 

Clone the repository and install dependencies:

```bash
git clone <your-repo-url>
cd llm-wiki
python -m venv .venv

# Windows
.venv\Scripts\activate
# Mac/Linux 
source .venv/bin/activate

# Install core dependencies
pip install -r requirements.txt

# Install graph dependencies (required for tools/build_graph.py)
pip install networkx python-louvain
```

### 3. Configuration

By default, the system uses **Gemini**. Copy the `.env.example` file to `.env` and insert your API key:

```bash
cp .env.example .env
```
Add your Gemini Key inside `.env`:
```env
GEMINI_API_KEY=your_google_api_key_here
```

*Note: All core paths (like the Obsidian vault destination and sources folder) and LLM Provider limits can be fine-tuned inside `config.json`.*

## 📖 Usage

LLM Wiki is built via distinct CLI entry points inside the `tools/` folder.

### Ingestion

Ingest a URL, PDF, or text file. This will initiate the batch review cycle where the LLM proposes wiki pages, which you must explicitly approve or edit.

```bash
python tools/ingest.py "https://example.com/article"
python tools/ingest.py ./sources/documents/research_paper.pdf
```

### Knowledge Graph Validation

Rebuild the offline interactive Knowledge Graph (`vault/graph/graph.html`). The crawler builds explicit wikilinks and invokes the LLM to map hidden semantic relationships.

```bash
# Full rebuild with semantic inference 
python tools/build_graph.py

# Structural-only rebuild (bypasses the LLM, much faster)
python tools/build_graph.py --no-infer

# Open the graph in your browser automatically
python tools/build_graph.py --open
```

### Health Check (Linting)

Ensure your offline wiki has no broken wikilinks or orphaned pages.

```bash
# Run health check
python tools/lint.py

# Save the results to vault/wiki/lint-report.md
python tools/lint.py --save
```

### Querying

Query your generated wiki natively in natural language.

```bash
python tools/query.py "What does the wiki say about attention mechanisms?"
```

## 📁 Architecture 

Sources must be kept entirely separated from the vault!

```text
project-root/
├── tools/                 # CLI Command execution points
├── config.json            # Configuration and Prompt boundaries
├── .env                   # Secret API Keys
├── sources/               # Raw sources (PDFs, text chunks) -> READ ONLY
└── vault/                 # The actual generated Obsidian Vault
    ├── wiki/
    │   ├── index.md       # Content catalog
    │   ├── entities/
    │   └── concepts/
    └── graph/             # Built UI dependencies
        └── graph.html   
```
