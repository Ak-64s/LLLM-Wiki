# S-8.5 Ingest CLI Orchestration — Pre-Implementation ADR

## Goal
Wire the isolated domain modules (S-1 config/provider → S-2 extract → S-3/4 plan & batch → S-5 interactive review → S-6 edit prop → S-7 atomic commit) into the primary executable `tools/ingest.py`. The interactive user must be able to run `python tools/ingest.py <source>` and navigate the entire pipeline as naturally intended by the project brief.

## Systems Touched
- `tools/ingest.py` — The thin CLI parsing interface wrapper.
- `src/core/ingest.py` — **New** domain orchestrator module abstracting cross-boundary logic.
- `tests/unit/test_s8.5_ingest.py` — Integration level tests mocking endpoints to trace orchestrator paths.
- `BACKLOG.md` — Slice status syncing.

## Assumptions
- `load_config`, `extract_source`, `complete`, `review_batch`, `propagate_edits`, and `commit_approved_batch` currently support native procedural bindings implicitly without concurrency or async/await overhead constraints.
- Any unhandled system failure generated within core operations (like `TesseractNotFoundError` or `SystemExit`) natively provides sufficient trace-text to output cleanly at the orchestration root without requiring complex retry mapping blocks (per AD-9 provider retries already established inside S-1).

## Constraints
- **C-1 (Python only)**: Requires standard execution routing.
- **C-9 (System editor)**: Must correctly bubble `$EDITOR` control from the review loops back up.
- Code blocks must provide clear `stdout` checkpoints so the executing CLI user understands where inference or delays are occurring between steps (e.g. "Drafting pages..." -> "Opening Review Loop...").

## Decision: Architecture Approach

We evaluated 2 distinct approaches for the orchestrator implementation:

**Approach A (Fat Orchestrator):** 
Code the procedural pipeline directly inside `tools/ingest.py`. 
*Pros*: Exposes standard procedural tracing inside one file. 
*Cons*: Completely violates structured domain logic established in S-1 -> S-8. Hard-wiring integration logic into a CLI executable makes unit testing impossible without aggressively patching `sys.argv` and `sys.stdout`.

**Approach B (Thin CLI + Core Orchestrator):** 
Generate `src/core/ingest.py` deploying an `orchestrate_ingestion(source: str)` function. `tools/ingest.py` will handle only argument parsing, error trapping bounding (`except SystemExit:`), and invoking the runner.
*Pros*: Follows `ENGINEERING.md` bounds. Allows injecting mock configurations and mock LLMs directly at python suite execution to test bridging integration paths cleanly. 

**Decision**: Chosen **Approach B**. S-8.5 deploys `src/core/ingest.py` for structured pipeline traversal.
