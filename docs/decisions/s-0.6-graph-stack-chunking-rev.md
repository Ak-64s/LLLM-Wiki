# ADR: Graph Stack & Chunking Thresholds

| Field   | Value                                          |
|---------|-------------------------------------------------|
| Slice   | S-0.6                                           |
| Date    | 2026-04-12                                      |
| Status  | ACCEPTED                                        |
| Decides | AD-12 (graph stack), AD-5 (chunking thresholds) |

---

## Context

Before implementation begins at S-1, two cross-cutting decisions must be locked:

1. **Graph stack** — which libraries build, cluster, and render the wiki knowledge graph
   (`build_graph.py`, S-9).
2. **Chunking thresholds** — when to split source text into chunks before sending to the LLM
   (LLM layer, S-3).

Both decisions propagate into `config.json` schema, dependency list, and the contracts for
S-3 and S-9. Changing either after implementation starts would require rework across multiple
slices.

**Prior art:** SamurAIGPT/llm-wiki-agent `build_graph.py` uses NetworkX + vis.js for graph
construction and rendering. SHA256 caching pattern borrowed from the same repo.

**Binding constraints:**
- C-1: Python only for core tool.
- C-10: LM Studio threshold 4,000 tokens; Gemini threshold 750,000 tokens; both configurable.
- C-12: vis.js loaded from CDN in graph.html. No npm, no build step.
- C-13: graph.html must be self-contained — readable offline after initial CDN load.

---

## Decision A: Graph Stack (AD-12)

### Approach 1 — NetworkX + python-louvain + vis.js (CDN) [CHOSEN]

| Component      | Role                      | Why this one                                          |
|----------------|---------------------------|-------------------------------------------------------|
| NetworkX       | Graph construction        | Pure Python. No compiled deps. Mature API. Handles <300 nodes trivially. |
| python-louvain | Community detection       | Single-purpose Louvain implementation. One function call: `community.best_partition(G)`. Deterministic. |
| vis.js (CDN)   | Interactive visualization | Zero build step. Built-in physics layout. ~10 lines of config for a working interactive graph. Self-contained HTML output. |
| SHA256 cache   | Incremental rebuild       | Hash per wiki page stored in `graph.json`. Only changed pages reprocessed. |

Edge types produced:
- `EXTRACTED` — explicit `[[wikilink]]` found in page content.
- `INFERRED` — LLM-detected implicit relationship, stored with confidence score 0–1.
- `AMBIGUOUS` — inferred edge below configurable confidence threshold.

Trade-off accepted: vis.js requires internet on first load (L-10). After initial CDN fetch,
the browser caches the script and graph.html works offline (C-13 satisfied).

### Approach 2 — igraph + D3.js (bundled) [REJECTED]

| Component | Role                      | Why rejected                                          |
|-----------|---------------------------|-------------------------------------------------------|
| igraph    | Graph construction        | Faster algorithms (C-compiled), but adds compiled dependency. Install friction on Windows (requires C compiler or pre-built wheel). Violates the spirit of C-1's simplicity. |
| D3.js     | Interactive visualization | Full control over rendering, but no built-in physics layout comparable to vis.js. Requires ~200+ lines of custom force-layout code vs. ~10 lines of vis.js config. |

Bundling D3.js (~250KB minified) into the HTML would eliminate the CDN dependency, but the
additional rendering code surface area and maintenance burden outweigh the offline-first
benefit — especially since vis.js CDN with browser caching already satisfies C-13.

### Why Approach 1 wins

1. **Pure Python install.** `pip install networkx python-louvain` — no C compiler, no platform-
   specific wheel hunting. Aligns with C-1.
2. **Minimal rendering code.** vis.js physics graph with interactive drag, zoom, and hover
   works out of the box. D3 would require writing and maintaining a custom force simulation.
3. **Community detection is one call.** python-louvain wraps the Louvain algorithm in a single
   function. igraph's community detection API is more powerful but unnecessary at <300 nodes.
4. **Reference validation.** SamurAIGPT/llm-wiki-agent already proves this stack works for
   wiki-scale knowledge graphs.

## Decision B: Chunking Thresholds (AD-5)

### Approach 1 — Provider-specific thresholds [CHOSEN]

| Provider  | Threshold     | Rationale                                                |
|-----------|---------------|----------------------------------------------------------|
| LM Studio | 4,000 tokens  | Qwen 3.5 9B has a limited, unreliable context window (A-3 UNTESTED). Conservative threshold avoids garbled output. Chunking is effectively always-on for non-trivial sources. |
| Gemini    | 750,000 tokens| 1M token context (C-4). 750k leaves 250k headroom for system prompt, index.md, and wiki page context. Forcing chunking on small sources wastes API calls and degrades synthesis coherence. |

Both thresholds stored in `config.json` as `chunk_threshold_lmstudio` and
`chunk_threshold_gemini`. User can tune as provider capabilities change or as A-3 is tested.

Chunk boundaries overlap to mitigate seam loss (L-2). Overlap size is also configurable.

### Approach 2 — Unified threshold [REJECTED]

A single `chunk_threshold` value applied to both providers.

**Why rejected:** The providers have a 187x difference in context capacity (4k vs. 750k).
A unified threshold forces one of two bad outcomes:
- Set to 4k: Gemini chunks unnecessarily. A 50k-token article becomes 12+ API calls instead of
  one. Synthesis coherence degrades. API quota consumed faster (L-8).
- Set to 750k: LM Studio receives inputs far beyond its context window. Output is garbled or
  truncated with no error signal — a silent failure (violates ENGINEERING.md: "never swallow
  errors").

Neither compromise is acceptable.

### Why Approach 1 wins

1. **Matches reality.** Each provider has fundamentally different capabilities. The config
   should reflect that, not paper over it.
2. **Trivial config cost.** Two JSON fields instead of one. No code complexity increase —
   the chunking engine reads `config[f"chunk_threshold_{provider}"]`.
3. **Future-proof.** When A-3 is tested and Qwen proves reliable at higher context, the user
   adjusts one field. No code change needed.

---

## Consequences

1. **S-9 (Graph Tool)** is locked to NetworkX + python-louvain + vis.js. No alternative
   evaluation needed at implementation time. Dependencies: `networkx`, `python-louvain`.

2. **S-3 (LLM Layer)** must branch on provider when deciding whether to chunk. The chunking
   engine reads the provider-specific threshold from config. Two threshold values, one code
   path with a parameterized threshold.

3. **config.json schema** must include:
   - `chunk_threshold_lmstudio` (default: 4000)
   - `chunk_threshold_gemini` (default: 750000)
   - `graph_infer_confidence_threshold` (default: TBD in S-9)

4. **Untested assumptions** remain open:
   - A-3: Qwen 3.5 9B coherence at 4k chunks — tested in S-3.
   - A-11: NetworkX + python-louvain cluster quality at <300 pages — tested in S-9.
   - A-12: LLM-inferred edge precision — tested in S-9.

## ENGINEERING.md Checklist

| Principle              | Application                                                      |
|------------------------|------------------------------------------------------------------|
| API & Contract Design  | graph.json schema and chunking I/O contracts are implicit here; formalized in S-3/S-9 contracts. |
| Performance            | SHA256 cache avoids repeated graph work. `--no-infer` flag controls expensive LLM calls. Provider-specific thresholds prevent unnecessary chunking on Gemini. |
| Integration            | Graph builder reads wiki pages (boundary I-7) and calls LLM (boundary I-8). Both validated at those boundaries. |
| Error Handling         | Inference failure per-page is non-fatal (graph build continues with structural edges). Chunk seam loss is a known limitation (L-2), not a silent failure. |
| State Management       | `graph.json` is the single source of truth for graph cache. Chunk thresholds have a single source in `config.json`. |
| Observability          | Graph builder logs which pages were reprocessed vs. cache-hit. Chunking engine logs chunk count and overlap size. |
